## Per-Scenario Evidence

### E2E-01 — Contiguous prune success (PASS-via-deterministic)

The mcp-server suite covers the deterministic resolver and write path:

- `resolveUniqueBoundary enforces unique match` — verifies that a unique pattern resolves to a single message; ambiguity is rejected upstream.
- `prune prewrite path does not mutate marker state` — verifies that the prune execution sequence performs argument validation BEFORE any JSONL or marker-file mutation, so failures roll back atomically.
- `encodes prune/retrieve success metadata in tagged base64 blocks` — verifies the success-response payload shape that downstream callers (and the model, via `tool_result`) consume.

These do not exercise a live `~/.claude/projects/<hash>/<sid>.jsonl` round-trip — that's E2E-01's live form. For v0.1.0 the deterministic suite is the documented evidence.

### E2E-02 — Boundary ambiguity (PASS-via-deterministic)

The "prune call filtering" suite covers both the genuine ambiguity-rejection path AND the wrapper-filter retry:

- `resolves to one non-wrapper candidate among wrapper candidates` — wrapper-filter returns the survivor.
- `throws ambiguity when multiple non-wrapper candidates remain` — filter→>1 behaves as plain ambiguity.
- `throws ambiguity when only wrapper candidates match` — filter→0 keeps the original error.
- `returns a single match without filtering` — single-match path is untouched.

The "loadSearchableMessages prune tool detection" suite verifies the predicate that powers the filter:

- `flags Claude Code MCP prune tool-use wrappers` (`mcp__context-bonsai__context-bonsai-prune`)
- `flags bare compatibility prune tool-use wrappers` (`context-bonsai-prune`)
- `does not flag non-prune assistant tool-use wrappers`
- `does not flag user tool-result-only messages`

### E2E-03 — Retrieve by anchor success (PASS-via-deterministic)

- `retrieve args accept anchor_id only` — verifies the input contract.
- `retrieve post-mutation cleanup failure still reports success` — verifies that a partial cleanup failure (e.g. marker-file unlink failure after JSONL restoration) does not falsely report failure when the user-visible state is correct.

### E2E-05 — Compatibility error path (PARTIAL-via-deterministic)

- `rejects ID selector patterns` and `rejects ID selector fields on prune` cover the schema-validation class of compat errors.
- The missing-JSONL / corrupted-JSONL paths are not deterministic-tested in v0.1.0; manual verification per `docs/e2e-protocol.md` E2E-05.

### E2E-04, E2E-06, E2E-07 — DEFERRED

These require:

- E2E-04 (gauge cadence): driving enough turns in a live Claude Code session to cross gauge severity thresholds, then inspecting the tool-response body or tweakcc UI.
- E2E-06 (persistence): exiting Claude Code and resuming via `claude --resume <session-id>` to verify archived state survives.
- E2E-07 (secret oracle): driving Claude through a seeded-secret + prune + secret-recall cycle to verify the model cannot recover the pruned content.

All three require interactive driving against a separate Claude Code instance. They are documented in `docs/e2e-protocol.md` and SHOULD be run before any tagged release beyond v0.1.0. The v0.1.0 release ships the protocol document so users can run these themselves.
